# 前端平台移植文档 (rmp_ui_test to source)

本文档旨在为前端项目从旧平台 `rmp_ui_test` 迁移至新平台 `source` 提供技术分析和实施指导。

## 1. 总结

- **旧平台 (rmp_ui_test)**: 基于 **UmiJS** 框架（`@oula/oula` 定制版），内置了路由、构建和状态管理（`dva`）。这是一个高度集成的“全家桶”式解决方案。
- **新平台 (source)**: 基于 **Vite + React**，这是一个更为灵活和现代化的组合。使用 **Vite** 作为构建工具，**React** 作为核心库，并依赖 **react-router-dom** 进行路由管理。

核心移植工作是从 UmiJS 的约定式、配置式开发模式，迁移到 Vite + React 的模块化、显式配置的开发模式。重点在于**路由迁移**、**状态管理重构**和**数据请求适配**。

---

## 2. 架构差异分析

| Aspect | 旧平台 (rmp_ui_test) | 新平台 (source) | 差异说明与影响 |
| :--- | :--- | :--- | :--- |
| **构建与开发** | UmiJS (`@oula/oula`) | Vite (`vite`) | **核心差异**。UmiJS 封装了 webpack，提供了大量内置功能。Vite 则以其极快的冷启动和热更新速度著称。开发和构建命令完全不同。 |
| **框架** | React 18 + UmiJS | React 18 (版本更新) | 基础库一致，但上层框架不同。UmiJS 有自己的渲染和路由入口，Vite项目则需要手动设置。 |
| **路由管理** | UmiJS 路由 (`@umijs/route-utils`) | React Router v6+ (`react-router-dom`) | UmiJS 通常使用配置文件（如 `config/routes.ts`）自动生成路由。React Router 需要在代码中通过 JSX（`<Routes>`, `<Route>`）或配置对象来显式定义。 |
| **状态管理** | Dva.js | React Context (推断) | **重大重构点**。`dva` 是基于 redux-saga 的数据流方案，有 `models` 的概念。新平台从文件结构看，倾向于使用 React 原生的 `Context API`。从 `dva` 到 `Context` 的迁移是本次移植的主要工作量之一。 |
| **数据请求** | `umi-request` | 原生 `fetch` 或 `axios` (未在 `package.json` 中明确) | 两个项目都在 `src/utils/request.ts` 中封装了请求逻辑。这降低了迁移成本，但仍需适配底层实现和拦截器逻辑。 |
| **样式方案** | Less | Less + Tailwind CSS | 新平台增加了对 `Tailwind CSS` 的支持。现有的 Less 样式文件可以迁移，但需要确保 Vite 的 Less 处理器配置正确。新组件可能会使用 Tailwind，需要注意样式一致性。 |
| **代理配置**| `config/proxy.ts` | `vite.config.ts` 中的 `server.proxy` | API 代理配置的位置和语法不同，需要将旧配置翻译成 Vite 的格式。 |

## 3. 移植步骤建议

### 3.1. 路由移植

1.  **分析旧路由**: 打开 `rmp_ui_test/config/routes.ts`，理解现有的路由结构、布局（Layout）、权限和嵌套关系。
2.  **创建新路由配置**: 在 `source/src/` 目录下，参照 `react-router-dom` 的 `createBrowserRouter` API，将旧的路由配置翻译成新的路由对象数组。特别注意 `element` 属性，它需要直接指向页面组件。
    -   UmiJS 的 `component` 路径字符串需要替换为 `import` 的组件。
    -   UmiJS 的 `routes` 嵌套数组需要转换为 `children` 属性。
3.  **渲染路由**: 在 `source/src/main.tsx` 文件中，使用 `createBrowserRouter` 和 `<RouterProvider>` 来启动应用和渲染路由。

### 3.2. 后端接口调用 (数据请求层)

1.  **统一代理配置**:
    -   打开 `rmp_ui_test/config/proxy.ts`。
    -   打开 `source/source/vite.config.ts`。
    -   将旧的代理规则（`target`, `changeOrigin`, `pathRewrite`）迁移到 `vite.config.ts` 的 `server.proxy` 对象中。语法基本兼容。

2.  **适配 `request.ts`**:
    -   对比 `rmp_ui_test/src/utils/request.ts` 和 `source/source/src/utils/request.ts`。
    -   **核心目标**: 确保两个`request`函数（或导出的HTTP方法）的**调用签名和返回格式**（Promise resolve/reject 的数据结构）保持一致。
    -   检查并迁移请求拦截器（如添加认证 `token` 的逻辑）和响应拦截器（如统一错误处理逻辑）。

### 3.3. 状态管理重构 (Dva to React Context)

这是工作量最大的部分，建议按页面或功能模块逐步进行。

1.  **创建 Context**: 针对原 `dva` model 管理的每一个全局或领域状态（如用户信息 `userInfo`），在 `source/src/store` 或 `source/src/contexts` 目录下创建一个新的 Context。
    ```tsx
    // 例如: source/src/contexts/UserContext.tsx
    import { createContext, useContext, useState } from 'react';

    const UserContext = createContext(null);

    export const UserProvider = ({ children }) => {
      const [user, setUser] = useState(null);
      // 可在此处添加 login, logout 等方法
      const value = { user, setUser };
      return <UserContext.Provider value={value}>{children}</UserContext.Provider>;
    };

    export const useUser = () => useContext(UserContext);
    ```

2.  **包裹根组件**: 在 `source/src/App.tsx` 或 `source/src/main.tsx` 中，用创建的 `Provider` 包裹整个应用或相关部分。

3.  **重构组件**:
    -   找到使用 `dva` `connect` HOC 或 `useDispatch` / `useSelector` 的组件。
    -   移除 `dva` 的连接代码。
    -   在组件内部使用对应的 `useContext` hook (`useUser()`) 来获取状态和修改状态的方法。
    -   原先调用 `dispatch` 触发 `effects` (异步操作) 的地方，需要调用从 Context 获取的相应方法（该方法内部执行异步请求）。

### 3.4. 组件与页面移植

1.  **复制文件**: 将 `rmp_ui_test/src/pages` 和 `rmp_ui_test/src/components` 的文件物理复制到 `source/source/src/pages` 和 `source/source/src/components`。
2.  **修复路径**: UmiJS 常用的 `@/` 别名在 Vite 中也需要配置（在 `vite.config.ts` 的 `resolve.alias` 中）。确保路径解析正确。
3.  **代码适配**:
    -   **Umi API 替换**: 替换 UmiJS 的特有 API，例如：
        -   `history.push('/path')` -> 使用 `react-router-dom` 的 `useNavigate` hook: `const navigate = useNavigate(); navigate('/path');`
        -   `@umijs/max` 提供的 `useAccess` 等权限相关API，需要根据新平台的权限方案进行重写。
    -   **样式导入**: 确保 `.less` 或 `.css` 文件在新组件中被正确 `import`。
    -   **状态管理**: 对每个组件执行 **步骤 3.3** 的重构。

### 3.5. 静态资源和环境变量

1.  **静态资源**: 将 `rmp_ui_test/public` 和 `rmp_ui_test/src/assets` 下的图片、字体等资源，复制到 `source/source/public` 或 `source/source/src/assets` 目录。
2.  **环境变量**: UmiJS 的环境变量（`.env` 文件和 `config.ts` 中的 `define`）需要迁移到 Vite 的模式。Vite 要求客户端暴露的环境变量必须以 `VITE_` 为前缀。

## 4. 风险和注意事项

- **工作量评估**: 状态管理 (`dva` -> `Context`) 是最大的技术债，迁移工作量巨大且容易出错，需要细致规划。
- **渐进式迁移**: 建议不要一次性迁移所有页面。可以选择一个相对独立的页面作为试点，跑通整个迁移流程（路由、状态、请求、样式），然后再逐步推广到其他页面。
- **第三方库兼容性**: 检查 `rmp_ui_test` 中使用的特殊第三方库，确认它们与 Vite 的兼容性。大部分库是通用的，但与构建工具深度绑定的插件可能需要寻找替代品。
- **测试**: 每个页面或组件迁移后，必须进行全面的功能回归测试，确保业务逻辑和UI表现与旧平台一致。
